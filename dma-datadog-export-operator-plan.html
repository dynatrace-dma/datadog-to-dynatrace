Operator Runbook · Dual-Platform

DMA DataDog Export — Operator Plan

Covers both macOS/Linux dma-datadog-export.sh and Windows dma-datadog-export.ps1 · v2.0.2 · Prepared 2026-06-03
Goal: produce one complete, verified .tar.gz export of a DataDog org for ingestion into the DMA tooling — on the first real run. The biggest failure mode is not a crash: it is an export that completes "successfully" but is silently missing data because the Application Key lacked a scope (DataDog returns 200 OK with empty results on most missing scopes). Both scripts are feature-equivalent for the core export — and now also for the concurrent detail-fetch and the best-effort Additional Resources pass (~25 extra config endpoints) — and share this plan; mac/Linux and Windows tags mark where commands or behavior differ.
The spine (8 phases, 3 gates):
  1. Phase 0 — Workstation & inputs ready (per-platform tooling)
  2. Phase 1 — Create credentials with the full scope set ← root cause of most bad exports
  3. Phase 2 — Determine region / site
  4. Phase 3 — Access test  GATE 1: zero FAILs (read the counts!)
  5. Phase 4 — Decide scope (skips, usage analytics, output)
  6. Phase 5 — Run the export
  7. Phase 6 — Verify completeness  GATE 2: no silent-failure box + manifest sane
  8. Phase 7 — Usage analytics + manual UI supplements (if in scope)
  9. Phase 8 — Hand off the archive  GATE 3: checksum verified

0Workstation & inputs

Tooling — differs by platform

macOS / Linux — needs curl, awk, tar (script aborts on startup if any is missing). All three ship with macOS and are standard on Linux — nothing to install (as of v2.0.2 the script is jq-free; JSON is parsed in pure bash/awk).
curl --version
awk  --version      # any POSIX awk (BSD awk on macOS, gawk/mawk on Linux)
tar  --version
bash --version      # 3.2+ is fine (macOS default)
Windows — zero external dependencies: PowerShell 5.1+ (pre-installed) and tar.exe (built into Windows 10 build 1803+ / Windows 11). No curl or jq needed.
$PSVersionTable.PSVersion    # expect 5.1+
tar --version                # built-in on Win10 1803+

Gather these before you start (both platforms)

Make the script runnable (one-time)

macOS / Linux
cd datadog-to-dynatrace
chmod +x dma-datadog-export.sh
If the file was edited/downloaded on Windows and you see $'\r': command not found, just run it once — it auto-converts its own CRLF line endings and re-executes.
Windows — Windows blocks unsigned scripts by default, and files downloaded from the web are "blocked".
cd datadog-to-dynatrace
Unblock-File .\dma-datadog-export.ps1
# then run with a per-process bypass (does not change machine policy):
powershell -ExecutionPolicy Bypass -File .\dma-datadog-export.ps1 -TestAccess
Or, in the session: Set-ExecutionPolicy -Scope Process Bypass then call .\dma-datadog-export.ps1 directly.

1Credentials & scopes — the critical step (identical on both platforms)

Why this is the whole ballgame: many DataDog read endpoints return 200 OK with an empty list when the Application Key lacks the scope — not a 403. So an export can finish green while silently dropping every dashboard, monitor, or SLO. The /api/v1/validate endpoint only checks the API Key, never the Application Key — so "credentials valid" tells you nothing about scopes. Treat the scope list below as mandatory.

Required Application Key scopes (always)

ScopeUnlocksIf missing
dashboards_readDashboards0 dashboards
monitors_readMonitors / alerts0 monitors
logs_read_configLog pipelines & indexes0 pipelines
synthetics_readSynthetic tests0 synthetics
slos_readSLOs0 SLOs
metrics_readMetric metadata (active, last 24h)0 metrics
monitors_downtimeDowntimes0 downtimes
integrations_readWebhook integrations0 webhooks
user_access_readUsers & roles0 users/roles
teams_readTeams0 teams
org_managementOrg name / metadataOrg shows "Unknown" in manifest

Extra scopes — only if running usage analytics

ScopeUnlocks
audit_trail_readMonitor modifications (both); monitor triggers + dashboard-view events mac/Linux
usage_readPer-index log ingestion volume (Usage Metering v1)

These fail silently to empty analytics files if absent — no error. Windows collects monitor triggers via the Events API instead of Audit Trail, so the API key also needs Events read for that one file (see Phase 7).

Scopes for the Additional Resources pass (best-effort — do not chase these)

Both scripts also sweep ~25 extra single-call config endpoints (notebooks, powerpacks, log archives/metrics, APM retention filters, spans metrics, RUM apps, security monitoring rules, service catalog, reference tables, incidents, AWS/Azure/GCP/PagerDuty integrations, host tags, …). Each touches its own read scope (e.g. apm_read, security_monitoring_rules_read, incident_read, integration scopes).

Operationally these are non-fatal by design. A missing scope (401/403) or an endpoint your org doesn't use (404) is logged and skipped — it does not fail the export, does not trip the silent-failure detector, and is not counted in the manifest. Treat skipped — missing scope / not available lines for these as informational; only chase one if migration scoping specifically needs that resource.

Dedicated-org trap (both): some "dedicated" instances (custom domain like hxp.datadoghq.com) are isolated orgs on shared US1 infrastructure, not separate clusters. If hxp fails to connect, use site app — the API is the standard api.datadoghq.com. An unknown short code does not abort; the script warns and builds a guessed URL that then fails on every call. Windows-only: if a dedicated cluster's TLS cert isn't trusted by the Windows cert store (corporate CA / hostname mismatch), add -SkipCertCheck — trusted networks only. (PowerShell also force-enables TLS 1.2, since PS 5.1 defaults to 1.0/1.1.)

3Access test  GATE 1

Always run the access test before the first real export. It probes all export categories plus the 2 analytics APIs, writes nothing, prompts for any missing keys, and prints a results table. It always exits 0 on both platforms — so in automation, parse the output, don't trust the exit code.

macOS / Linux
./dma-datadog-export.sh --test-access                    # prompts for keys + site
./dma-datadog-export.sh --api-key "<API>" --app-key "<APP>" --site app --test-access
Windows
.\dma-datadog-export.ps1 -TestAccess
.\dma-datadog-export.ps1 -ApiKey "<API>" -AppKey "<APP>" -Site app -TestAccess

Reading the result — and a key platform difference

ResultMeaningAction
PASSmac/Linux 200 and at least one item found.
Windows 200 — even with 0 items (shows "N found").		Good on mac/Linux. On Windows, read the count in the Detail column — a green "PASS — 0 found" for a category you expect to have data is a red flag.
WARNmac/Linux 200 but 0 items, or 404.
Windows optional/auth issue on analytics, Webhooks, or Teams.		OK only if you genuinely expect that category to be empty / aren't running --usage.
FAIL401/403 (scope/auth), no response, or unexpected code.Must fix before exporting.
GATE 1 — do not proceed until: zero FAIL rows, and every category you expect to contain data shows a non-zero count (on Windows verify the count, since 0-items still reads PASS). If usage analytics are planned, the two analytics rows must be PASS/WARN, not FAIL. Fix by re-creating the Application Key with the missing scopes (Phase 1) and re-testing.
Quirks in the access test itself:
  • Both: exit code is always 0, even with FAIL rows — parse output for FAIL in CI.
  • mac/Linux only: the Webhooks and Teams probes hit different endpoints than the real export, so an odd WARN on just those two isn't predictive — confirm them in the manifest after the run. (On Windows these probes match the export endpoints, so they're trustworthy.)
  • Windows: if Credentials FAIL, the table aborts early and no other rows are shown — fix the keys first.

4Decide scope

Decision.sh flags.ps1 paramsGuidance
What to collect--skip-dashboards…--skip-users-SkipDashboards…-SkipUsersDefault is everything. Downtimes, webhooks, and the Additional Resources pass always run (no skip). Use skips to re-run one slow category or work around a scope you can't get yet.
Concurrency capsDASHBOARD_CONCURRENCY (10) · SYNTHETICS_CONCURRENCY (10) · LOGS_CONCURRENCY (5) — env vars, both platformsDashboards/pipelines/synthetics details are fetched concurrently (bash curl --parallel; PowerShell runspace pool). Lower these if you hit sustained 429s on a constrained org.
Usage analytics--usage / --usage-period 90d-Usage / -UsagePeriod 90dOff by default. Adds the 2 audit/usage scopes and dominates runtime. See Phase 7.
Output--output DIR / --name N-Output DIR / -Name NDefault ./datadog-export + timestamped name. Point at a disk with 500 MB+ free.

Runtime ballpark (no usage): small org 2–5 min, medium 10–20 min, large 30–60 min. With usage analytics, roughly 3–4× (Audit Trail pagination dominates). Detail-by-ID fetches (dashboards, pipelines, synthetics) run concurrently on both platforms, so large orgs finish much faster than a sequential pass; a 429 during the concurrent phase is retried with back-off, and list-level 429s are auto-retried (3 tries, 5/10/15s) — no action needed.

5Run the export

Interactive — recommended on shared hosts (keys aren't in shell history or ps). Both prompt only for values you don't supply.

macOS / Linux
./dma-datadog-export.sh
# prompts for API key, App key, and site; then runs
Windows
.\dma-datadog-export.ps1
# prompts for API key, App key, and site; auto-prompts to create the output dir if absent

Non-interactive — automation / repeatable:

macOS / Linux — supply all three values; there is no --non-interactive flag (passing it errors out).
./dma-datadog-export.sh \
  --api-key "<API>" --app-key "<APP>" \
  --site app \
  --output ~/dd-export \
  --usage --usage-period 90d
Windows — use the explicit -NonInteractive switch (it requires keys, and auto-creates the output dir).
.\dma-datadog-export.ps1 `
  -ApiKey "<API>" -AppKey "<APP>" `
  -Site app `
  -Output C:\dd-export `
  -Usage -UsagePeriod 90d `
  -NonInteractive
Credential hygiene (both): in non-interactive mode the keys are passed as process arguments — visible in process listings (ps aux / Task Manager / Get-CimInstance Win32_Process) and your shell history, and neither script reads keys from environment variables. On a shared or CI host, prefer interactive mode, or use an ephemeral runner and rotate the Application Key after the export. Never commit keys to the repo.

What you'll see (both): requirements check → credential validation (org name printed) → each core category (dashboards/pipelines/synthetics fetched concurrently, with a "max-parallel=N" line in export.log) → the Exporting Additional Resources step (a one-line PASS/skip per resource) → (usage analytics) → manifest → archive + checksum → an Export Summary. Individual category failures are logged and the run continues — so "it finished" is not the same as "it's complete." That's Phase 6. Note: skipped / not available lines during the Additional Resources step are expected best-effort skips, not failures (Phase 1).

6Verify completeness  GATE 2

Watch for the silent-failure banner (both platforms). If any tracked category returned 200-but-zero, the script prints a "⚠ POTENTIAL SILENT FAILURES DETECTED" box at the end listing each empty category and its likely-missing scope. If you see it, the export is almost certainly incomplete — fix the scope (Phase 1), re-test (Phase 3), re-run. The detector covers dashboards, monitors, log pipelines, synthetics, SLOs, metrics, and users — but not log indexes, downtimes, webhooks, roles, or teams, so still eyeball those counts in the manifest. The best-effort Additional Resources are deliberately outside the detector and the manifest exported_items totals — verify a specific one by listing its folder on disk (e.g. integrations/, apm/, security/) if migration scoping needs it.

Verification checklist

  1. No silent-failure box was printed (and the "Completed with N errors" count is acceptable).
  2. Inspect the manifest — the authoritative record. Confirm organization_name/id is the right org (correct keys/region), exported_items counts are non-zero where you expect data, and failed_api_calls is low.
    macOS / Linux
    cd <output>/datadog-export-<TIMESTAMP>
python3 -m json.tool manifest.json   # or: cat manifest.json   (jq optional, not required)
ls dashboards | head                 # expect dashboard-<id>.json files, not just _list.json
    Windows
    cd <output>\datadog-export-<TIMESTAMP>
$m = Get-Content manifest.json -Raw | ConvertFrom-Json
$m.datadog_info; $m.export_statistics; $m.exported_items
Get-ChildItem dashboards | Select-Object -First 5
  3. If usage analytics ran: open analytics/_summary.json and confirm the per-query counts aren't all zero; check no file is error-only (Phase 7).
GATE 2 — the export is "good" when: no silent-failure box, manifest org is correct, and every in-scope category has a plausible non-zero count with detail files on disk. Otherwise diagnose (Troubleshooting) and re-run.

7Usage analytics & manual supplements (only if usage enabled)

With usage analytics on, the script writes an analytics/ folder: monitor_triggers.json, monitor_modifications.json, log_index_volume.json, unused_monitors.json, unused_dashboards.json, dashboard_views.json, and _summary.json. Audit Trail is paginated (up to 20 pages × 1000 events), which is why usage runs are slow.

Dashboard views are unreliable — plan to collect them manually. The two scripts genuinely diverge here (the README's "feature-equivalent" claim does not hold for this one file):
  • mac/Linux queries the Audit Trail for "Dashboard Viewed" events and aggregates real view counts — but only if your DataDog plan records those events; otherwise the file is an empty array.
  • Windows always writes {"error":"not_available"} and never attempts it.
Either way, capture the Dashboards → Popular Dashboards list from the UI (screenshot — DataDog exposes no API/export for view counts) and share it with your DMA consultant.

A second divergence to be aware of: monitor triggers come from the Audit Trail on mac/Linux but the Events API on Windows. Both produce monitor_triggers.json; just note the Windows path needs Events read access, not strictly audit_trail_read. If a usage file is empty, gather the equivalent from the UI and hand it over with the archive:

If empty…Likely causeUI fallback
log_index_volume.json ("error": "Usage Metering API not available")missing usage_readLogs → "Log Management - Estimated Usage" dashboard, Indexed Logs section
monitor_triggers.jsonAudit/Events access missingEvents Explorer sources:monitors grouped by monitor

8Hand off the archive  GATE 3

Outputs land at <output>/<name>.tar.gz with a sibling <name>.tar.gz.sha256. Verify integrity before uploading.

macOS / Linux
cd <output>
shasum -a 256 -c datadog-export-<TIMESTAMP>.tar.gz.sha256    # expect: OK
Windows (no -c verify built in — compare hashes)
cd <output>
$expected = (Get-Content datadog-export-<TIMESTAMP>.tar.gz.sha256).Split(' ')[0]
$actual   = (Get-FileHash datadog-export-<TIMESTAMP>.tar.gz -Algorithm SHA256).Hash.ToLower()
if ($actual -eq $expected) { "OK" } else { "MISMATCH" }
GATE 3: checksum matches and the archive size is plausible for the org. Then upload to the DMA Curator Server (recommended — full migration planning) or the DMA DataDog App (ad-hoc/smaller). The embedded manifest.json tells DMA which script version produced it and which categories are present. (Both scripts' closing text says "DMA / DMP (DataDog Edition)" — confirm the exact target with your DMA contact.)

Known doc-vs-script discrepancies (so they don't trip you up)

  • Dashboard views diverge across scripts (documented in README.md and USAGE.md) — mac/Linux attempts real Audit Trail data, Windows hardcodes not_available. Treat as best-effort; collect manually (Phase 7).
  • Monitor triggers use different APIs per platform — Audit Trail (mac/Linux) vs Events API (Windows). Output file is the same; scope/access nuance differs.
  • --non-interactive does not exist in bash — passing it errors. Only PowerShell has a real -NonInteractive; for bash, just supply the three required values.
  • Access-test reading differs: mac/Linux downgrades a 200-with-0-items to WARN; Windows reports it as PASS "0 found." Always read the counts.
  • Access-test exit code is always 0 on both, and on mac/Linux the Webhooks/Teams probes use different endpoints than the export — verify those two in the manifest, not the access test.
  • Usage-analytics APIs differ by platform (now documented in README.md/USAGE.md): log index volume is Usage Metering v1 on mac/Linux vs v2 on Windows; monitor triggers come from Audit Trail (mac/Linux) vs the Events API (Windows). Functionally equivalent output.
